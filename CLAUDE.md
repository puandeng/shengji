# 200 Card Game (Sheng Ji)

## Objective
A real-time, browser-based multiplayer implementation of **200分 / Sheng Ji**, the Chinese trick-taking card game. Four players in two fixed teams compete across multiple rounds; the attacking team tries to collect ≥ 100 points worth of scoring cards (5s, 10s, Ks — 200 pts total in two decks) while the defending team tries to deny them. First team to win 3 rounds wins the match.

The goal of this project is to provide a clean, self-contained reference implementation playable over a LAN or the internet via a 4-letter room code — no accounts, no persistence, just open the page and play.

## Architecture
Monorepo with npm workspaces.

```
shengji/
├── server/   # Node.js + Express + Socket.io — authoritative game state
└── client/   # React 18 + Vite — UI and socket client
```

- **Server is authoritative.** All game logic (dealing, follow-suit validation, trick resolution, scoring) lives in `server/game/` and runs on the server. Clients only send intents (`game:playCard`, `game:declareTrump`, …) and receive state snapshots.
- **Transport:** Socket.io rooms keyed by a 4-letter code generated by `RoomRegistry` (`server/game/Room.js`).
- **State:** in-memory only — no database. Rooms are deleted when empty.
- **Per-player views:** `GameState.toPlayerJSON()` strips other players' hands before sending, so a client only ever sees its own cards plus opponents' hand counts.

|---|---|

| `server/socket/gameHandlers.js` | `game:declareTrump`, `game:discardKitty`, `game:playCard` |

## Game Flow
1. **WAITING** — players join via room code; host starts when 4 are seated.
2. **DEALING** — 2 decks (104 cards) → 25 per player + 4 kitty.
3. **TRUMP_SELECTION** — 30s window; any player may declare by playing a card whose suit becomes trump. On timeout, `autoSelectTrump()` picks from seat 0's hand. Declarer's team becomes attacker.
4. **KITTY** — declarer picks up the 4 kitty cards and discards 4 back. Discarded kitty points are doubled and awarded to whoever wins the **last** trick.
5. **PLAYING** — standard trick-taking, must follow lead suit if held. Trump beats non-trump; same group → higher rank wins.
6. **SCORING** — round ends when hands are empty. Attacker wins the round if they collected ≥ `WINNING_THRESHOLD` (100). Loser of the round attacks next.
7. **GAME_OVER** — first team to 3 round-wins takes the match.

## Game Rules Reference
| Rule | Value |
|---|---|
| Players | 4 (seats 0–3) |
| Teams | Seats 0&2 vs Seats 1&3 |
| Decks | 2 × 52 = 104 cards |
| Cards per player | 25 |
| Kitty size | 4 |
| Point cards | 5 → 5pts, 10 → 10pts, K → 10pts |
| Total points | 200 |
| Round threshold | 100 (attacker needs ≥ this to win the round) |
| Match length | First team to 3 rounds |
| Trump declare timer | 30s |

## Socket Protocol

### Client → Server
| Event | Payload | Description |
|---|---|---|
| `room:create` | `{ name }` | Create a new room |
| `room:join` | `{ name, code }` | Join existing room |
| `room:start` | — | Start the game (host only) |
| `room:chat` | `{ message }` | Send chat message |
| `game:declareTrump` | `{ cardId }` | Declare trump suit |
| `game:discardKitty` | `{ cardIds[] }` | Discard 4 cards to kitty |
| `game:playCard` | `{ cardId }` | Play a card |
| `room:newRound` | — | Start next round (host only) |

### Server → Client
| Event | Description |
|---|---|
| `player:joined` | A player joined the lobby |
| `player:left` | A player disconnected |
| `game:started` | Game has started (includes player's hand) |
| `game:trumpSelected` | Trump has been declared |
| `game:kittyDiscarded` | Kitty cards discarded, play begins |
| `game:cardPlayed` | A card was played (trick in progress) |
| `game:trickComplete` | Trick resolved (includes winner & scores) |
| `game:newRound` | New round started |
| `room:chatMessage` | New chat message |

## Conventions

### Code style
- **Indentation:** 4 spaces (note: existing files use 2 — match the surrounding file when editing)
- **Comments:** only when the "why" isn't obvious from the code
- **Server is authoritative** — never duplicate validation logic on the client; send an intent and trust the server response

### Collaboration (multiple Claude sessions / humans)
This repo is worked on by several people, often via parallel Claude Code sessions. Follow these rules to avoid stepping on each other:

1. **Always sync before starting work.**
   ```bash
   git checkout main && git pull --ff-only origin main
   ```
   Read `PLAN.md` to see what's already in flight before picking up a task.

2. **Never commit directly to `main`.** Always create a branch:
   ```bash
   git checkout -b feat/<short-description>
   ```
   Branch prefixes: `feat/`, `fix/`, `refactor/`, `docs/`, `test/`, `chore/`.

3. **Commits:** conventional commits (`feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`). Keep them small and focused; the commit body should explain *why*, not *what*.

4. **One task = one branch = one PR.** Don't bundle unrelated changes. If you discover a second issue, note it in `PLAN.md` and open a separate branch.

5. **Open a PR as soon as the branch has meaningful changes.**
   - Title: matches the conventional-commit style (`feat: add kitty animation`)
   - Body: what changed, why, how to test, and a link to the `PLAN.md` item it addresses
   - Mark as **Draft** if still in progress so others know not to review yet

6. **PRs require at least one approval before merging.** Do not self-merge unreviewed PRs. If a reviewer requests changes, push fixups to the same branch — don't open a new PR.

7. **Merge strategy:** squash-merge into `main`. Delete the branch after merging.

8. **Rebase, don't merge, when updating a branch from main:**
   ```bash
   git fetch origin && git rebase origin/main
   ```
   This keeps history linear. Force-push with `--force-with-lease` (never plain `--force`).

9. **Update `PLAN.md` as you go.**
   - Mark items in-progress when you start (add your name/session id)
   - Mark items done in the same PR that completes them
   - Add newly discovered todos to the bottom rather than silently dropping them

10. **Update `CLAUDE.md` when you change architecture, conventions, or the socket protocol.** If a future session would be misled by the current `CLAUDE.md`, fix it in the same PR. Treat `CLAUDE.md` as load-bearing documentation, not a README afterthought.

11. **Don't rewrite history on shared branches.** Rebasing your own unmerged feature branch is fine; rewriting `main` or anyone else's branch is not.

## Prerequisites
- Node.js 18+
- npm 9+

## Quick Start
```bash
# install everything (uses npm workspaces)
npm run install:all

# run server + client together
npm run dev
```

Or run them separately:
```bash
# Terminal 1
cd server && npm run dev    # → http://localhost:3001

# Terminal 2
cd client && npm run dev    # → http://localhost:5173
```

Then open `http://localhost:5173`, create a room, and share the 4-letter code with 3 friends.

### Environment
- `server/.env` — `PORT` (default 3001), `CLIENT_URL` (CORS origin, default `http://localhost:5173`)
- `client/.env` — server URL for the socket connection

### Build / Production
```bash
npm run build    # builds the client (Vite)
npm run start    # runs the server (node)
```
